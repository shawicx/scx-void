# 🚀 **AI Code Agent - Multi-Project CLI Rules**

以下规则适用于所有项目类型，包括：React、Vue、Node.js、Rust、Tauri、前端工具链类项目等。

---

## 1. **通用原则**

1. **除非用户明确要求，否则不得改变项目原有功能行为。**
   不得随意修改已有配置、接口行为、环境变量结构、目录结构、脚手架流程等。

2. **不得擅自生成总结文档、说明文档或 README 内容。**
   除非用户明确说需要写文档。

3. **禁止降级依赖版本。**
   只有在用户明确允许或必须降级以修复冲突时才可降版本。

4. **对新功能、文件、依赖的添加必须保持兼容性，**
   不得破坏项目现有逻辑、构建流程或运行结果。

5. **保持代码风格统一，文件命名、目录结构必须合理、可维护。**

---

## 2. **依赖与包管理规则（前端项目）**

1. **前端项目必须使用 `pnpm` 作为包管理器。**
   不得使用 npm、yarn、bun，除非用户主动说明例外。

2. **新增依赖必须遵循最新兼容版本策略（semver ^）。**

3. **不得移除现有依赖，除非用户明确要求或明确冗余。**

4. **修改 package.json 前需要考虑项目类型：**

   * React 项目遵循 JSX/TSX 标准
   * Vue 项目遵循 Vue 版本规范（Vue 2 / Vue 3 不可误配）
   * Node 项目保持 ESM/CJS 格式统一
   * Tauri 项目维护 Rust + 前端各自依赖的稳定性

---

## 3. **项目结构与文件规则**

1. 项目结构必须遵循「最小、清晰、可维护」原则。
   不能随意加入奇怪的目录或无意义的分层。

2. 不得重构项目一级目录结构（如 src、public、dist、cmd、apps 等），
   除非用户明确批准。

3. 添加文件必须遵守项目已有的组织方式：

   * React：组件放 components，hooks 放 hooks
   * Vue：组件 .vue 文件遵循已有分层
   * Node：遵循当前 CommonJS 或 ESM 模式
   * Rust：遵循 cargo 的模块系统（lib.rs / mod.rs）
   * Tauri：保持 src-tauri 规范结构

4. 文件名统一使用小写短横线（kebab-case）或遵循项目现有约定。

---

## 4. **跨项目的 CLI 工具设置**

AI 在执行 CLI 类自动化任务时需遵循：

1. **所有项目可共用的脚本必须设计为可插拔，不得强绑定框架。**

2. **生成的脚手架逻辑必须根据项目类型识别：**
   `react/vue/node/rust/tauri`
   严禁错误识别导致的配置污染。

3. Node CLI 的 bin 必须保持可执行、可移植，不得依赖项目路径外行为。

4. 提供的代码必须兼容主流运行环境：

   * 浏览器项目不得加入 node-only API
   * Node CLI 不得依赖浏览器 API（除非 polyfill 已明确存在）

5. 配置生成器不得覆盖已有配置，必须采取合并策略：

   * tsconfig.json → 合并 compilerOptions
   * vite.config / webpack → 合并插件
   * eslint → 扩展已有规则，不得重写
   * postcss / tailwind → 追加配置，不得破坏现有功能

---

## 5. **兼容性要求**

1. **浏览器项目必须保持浏览器兼容性**
   禁止插入 Node.js API（如 fs、path、process）。

2. **CLI 项目不得插入仅浏览器可运行的逻辑（如 DOM、window）。**

3. Rust / Tauri 项目：

   * 必须遵循 Rust 版本约束（edition 2018/2021）
   * Tauri 配置必须保留 security 相关字段

4. 跨项目模板生成器必须根据目标项目自动判断：
   不得混用环境。

---

## 6. **AI 自动修复、补全规则**

1. AI 在修补代码时必须尽量保持原代码风格与结构。
   不得把项目改成自己喜欢的风格。

2. 自动补全必须基于项目已有技术栈：
   react → 不写成 vue
   vue → 不写成 react
   node → 不加浏览器语法
   rust → 不生成 js 文件

3. **新增逻辑必须最小化修改范围**
   避免引起连锁兼容问题。

---

## 7. **输出格式要求**

1. 除非用户要求解释，否则输出仅提供代码或必要操作步骤。
2. 不得生成冗余描述性内容（如"总结如下""整体而言"等）。
3. 必要时才能提供路径结构、代码片段、命令行步骤。

---

## 8. **禁止事项**

以下行为全部禁止：

* 降级依赖
* 删除功能
* 重构项目大型结构
* 改变运行模式（dev → build）
* 修改 lint / build 行为导致结果不同
* 自动生成 README / docs（除非要求）
* 擅自改变 tsconfig / vite / webpack 的基础配置
* 将项目改为其他框架
* 插入使用不了的 API（例如 fs 到浏览器环境）
