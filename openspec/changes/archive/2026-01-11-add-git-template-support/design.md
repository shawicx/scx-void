# Git 模板支持设计文档

## 上下文

scx-void 当前支持本地模板初始化，用户只能从内置模板中选择项目类型。为了提供更灵活的项目初始化方式，需要添加从 Git 仓库获取模板的能力。

**约束**：
- 必须支持交互式和命令式两种使用方式
- 必须向后兼容现有本地模板功能
- 添加新模板类型不应修改核心逻辑
- 支持多种 Git 协议（HTTPS、SSH）
- 支持分支、标签和模板子路径选择

**利益相关者**：
- 开发者：需要快速初始化项目
- 团队：需要统一的团队模板
- 用户：需要自定义个人模板

## 目标 / 非目标

### 目标
1. 提供可扩展的 Git 模板下载系统
2. 支持预定义模板和自定义 Git 仓库
3. 实现双模式使用（交互式和命令式）
4. 提供良好的错误处理和用户反馈
5. 保持代码模块化和可测试性

### 非目标
- 不实现模板版本管理（由 Git 自身处理）
- 不实现模板更新检测（用户可重新克隆）
- 不实现模板配置文件（未来功能）
- 不实现模板市场或搜索功能

## 决策

### 1. 模块架构：分层设计

**决策**：采用分层架构，将 Git 模板功能划分为独立的模块

```
CLI 层（src/cli/project.rs）
    ↓
Services 层（src/services/project/project_service.rs）
    ↓
Git 模块（src/services/project/git/）
    ├── types.rs      # 数据结构
    ├── registry.rs   # 模板注册表
    ├── validator.rs  # 参数验证
    ├── clone.rs      # Git 克隆
    └── downloader.rs # 模板下载器
    ↓
Utils 层（src/utils/git.rs）
```

**原因**：
- 遵循项目现有分层架构
- 各模块职责清晰，易于测试和维护
- 便于未来扩展（如添加更多 Git 功能）

**考虑的替代方案**：
- 单文件实现：所有逻辑放在一个文件中
  - 拒绝原因：难以测试，违反单一职责原则
- 直接集成到现有模块：将 Git 功能添加到现有 service 模块
  - 拒绝原因：会增加现有模块的复杂度，难以独立测试

### 2. Git 命令执行：使用 duct 库

**决策**：使用 `duct` 库执行 Git 命令

**原因**：
- 项目已在其他地方使用 `duct`（系统关机功能）
- 提供跨平台支持
- 支持命令链和管道
- 易于测试（可模拟命令执行）

**考虑的替代方案**：
- 使用 `git2` crate：Rust 原生 Git 库
  - 拒绝原因：增加依赖复杂度，学习曲线陡峭
- 直接调用系统 `git` 命令：使用 `std::process::Command`
  - 拒绝原因：`duct` 提供更好的抽象和错误处理

### 3. 模板注册表：使用 LazyLock 静态注册

**决策**：使用 `std::sync::LazyLock` 实现静态模板注册表

**原因**：
- 编译时注册，运行时无性能开销
- 线程安全
- 简单直接，无需运行时配置
- 符合 Rust 惯用法

**考虑的替代方案**：
- 动态注册表：使用 `RwLock<HashMap>` 允许运行时注册
  - 拒绝原因：增加复杂度，当前不需要运行时注册
- 配置文件：从 `~/.scx-void/templates.toml` 读取
  - 拒绝原因：第一阶段不需要，可作为未来功能

### 4. 临时文件管理：使用 tempfile crate

**决策**：使用 `tempfile` crate 管理临时目录

**原因**：
- 自动清理（Drop 时删除）
- 跨平台支持
- 安全性（使用系统临时目录）
- 简化代码逻辑

**考虑的替代方案**：
- 手动管理临时目录：自己创建和删除
  - 拒绝原因：容易忘记清理，增加出错可能
- 使用固定临时目录：`/tmp/scx-void-temp`
  - 拒绝原因：并发问题，需要清理旧文件

### 5. CLI 参数设计：可选参数支持混合模式

**决策**：CLI 参数设计支持纯交互式、纯命令式和混合模式

**原因**：
- 提供最大的灵活性
- 满足不同用户偏好
- 向后兼容现有命令（不指定参数时交互式）

**参数设计**：
```rust
Init {
    /// 模板源 (local 或 git)
    #[arg(short, long)]
    source: Option<String>,

    /// Git 模板 ID（预定义模板的唯一标识）
    #[arg(short, long)]
    template: Option<String>,

    /// 自定义 Git 仓库 URL
    #[arg(short, long)]
    url: Option<String>,

    /// Git 分支或标签
    #[arg(short, long)]
    branch: Option<String>,

    /// 模板在仓库中的相对路径
    #[arg(long)]
    template_path: Option<String>,

    /// 项目名称
    #[arg(short, long)]
    name: Option<String>,
}
```

### 6. 错误处理：集中式错误类型

**决策**：在 `src/errors.rs` 中添加所有 Git 相关错误类型

**原因**：
- 保持项目现有错误处理模式
- 集中管理，易于维护
- 统一的错误展示格式

**新增错误类型**：
- `GitCloneError` - Git 克隆失败
- `InvalidGitUrl` - 无效的 Git URL
- `GitBranchNotFound` - Git 分支不存在
- `TemplateDownloadFailed` - 模板下载失败
- `GitTemplateNotFound` - Git 模板 ID 不存在
- `GitNotInstalled` - Git 命令未安装

## 风险 / 权衡

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| Git 命令未安装 | 功能无法使用 | 高 | 添加检测和友好的错误提示 |
| Git 仓库不存在或无权限 | 克隆失败 | 中 | 验证 URL 格式，捕获详细错误 |
| 网络问题 | 克隆超时或失败 | 中 | 使用浅克隆（depth=1）提高速度 |
| 临时目录清理失败 | 磁盘空间占用 | 低 | 使用 `tempfile` 自动清理，添加警告 |
| 模板路径错误 | 项目结构不正确 | 中 | 验证模板路径是否存在 |
| 大仓库克隆速度慢 | 用户体验差 | 中 | 默认使用浅克隆，提供进度提示 |

## 迁移计划

### 第一阶段：基础设施（无破坏性变更）
1. 创建 `src/services/project/git/` 模块结构
2. 实现数据结构定义（`types.rs`）
3. 实现模板注册表（`registry.rs`）
4. 实现 Git 工具函数（`utils/git.rs`）
5. 扩展错误类型（`errors.rs`）

### 第二阶段：核心功能（无破坏性变更）
1. 实现 Git 克隆功能（`clone.rs`）
2. 实现模板下载器（`downloader.rs`）
3. 集成到项目服务（`project_service.rs`）
4. 添加单元测试

### 第三阶段：CLI 集成（向后兼容）
1. 扩展 CLI 参数（`cli/project.rs`）
2. 实现交互式流程
3. 实现命令式流程
4. 添加集成测试

### 第四阶段：文档和优化
1. 更新 README 和使用文档
2. 优化错误消息和用户反馈
3. 性能测试和优化

**回滚计划**：
- 所有新功能都是增量添加
- 可通过删除 `src/services/project/git/` 目录回滚
- CLI 参数是可选的，不影响现有命令

### 测试依赖

- 需要测试并发克隆（如果支持）

## 性能考虑

1. **浅克隆**：默认使用 `--depth 1` 只克隆最新提交
2. **并发限制**：如果支持并发克隆，限制并发数（建议 3-5）
3. **缓存策略**：不实现缓存，避免复杂度
4. **进度提示**：对于大仓库，显示克隆进度

## 安全考虑

1. **URL 验证**：验证 Git URL 格式，防止注入攻击
2. **路径验证**：验证模板路径，防止路径遍历
3. **临时目录**：使用系统临时目录，避免敏感文件泄露
4. **权限检查**：检查项目目录写入权限
5. **Git 命令**：使用最小权限执行 Git 命令
